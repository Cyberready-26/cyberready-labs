DMZ — Web Layer
Firewall
Security checkpoint — allow or block every connection
First internal component a request reaches. Misconfiguration can block all customers or expose the internal network. Configuration changes are high-risk events requiring change management.
Firewall Concepts
1 / 5
Where the Firewall sits
Public Internet ends at the left. NorthBridge private network begins at the right. The Firewall is the gate between them.
One decision — Allow or Block
Every incoming connection is inspected. One of two outcomes. No middle ground.
Policies — the rulebook
Written by security engineers. Define exactly what is permitted and what is not. The Firewall enforces these rules automatically.
Ports — the server's doors
Every service uses a specific door (port). An attacker trying every door in sequence is port scanning. The Firewall detects this pattern and blocks it.
Attack signatures — known fingerprints
Every attack leaves a recognisable pattern. The Firewall checks every request against a database of known signatures and blocks matches automatically.

DMZ — Web Layer
Load Balancer
Distributes requests across multiple Web Servers
If the Load Balancer fails, no request can reach any Web Server — even though all servers are healthy. A single point of failure for all customer traffic passing through the Firewall.
Load Balancer Concepts
1 / 4
Where the Load Balancer sits
After the Firewall. Directly in front of the Web Servers. It decides which server handles each incoming request.
The problem — one server, thousands of requests
Without a Load Balancer, all traffic hits one server. At peak time — 9:15 AM Monday — it collapses under the load.
The solution — requests distributed evenly
Each server handles a share of the traffic. No single server is overwhelmed. If one slows down, the Load Balancer routes around it automatically.
Load Balancer failure — all traffic blocked
Even though all Web Servers are healthy, no request can reach them. The failure is in the routing, not the processing.

DMZ — Web Layer
Web Server
Bank's front door — receives and routes requests
Directly exposed to public internet. First application-layer system an attacker probes. A vulnerability here can give a foothold inside the bank's infrastructure.
Web Server Concepts
1 / 3
Web Server — the bank's front door
Like a reception desk: receives the request, does a quick check, and routes it to the right internal system. Does not process logic or hold data.
What the Web Server does NOT do
Each component in the chain does only its specific job. The Web Server receives and routes. Nothing more. This separation is intentional — it is part of defence in depth.
The most exposed component
The Web Server faces the public Internet directly. It is the first application-layer system an attacker reaches. A vulnerability here is a foothold into the bank.

Internal Secure Network
Application Server
Verifies identity and authorisation · business logic
Highest-value target. Handles all authentication and authorisation logic. Compromise gives access to every customer account. Not directly exposed to internet — protected by Firewall, Load Balancer, and Web Server layers.
App Server Concepts
1 / 3
Authentication vs Authorisation — two separate checks
Authentication confirms who you are. Authorisation confirms what you are allowed to access. The App Server performs both — in that order — for every request.
Service dependency in action
The App Server depends on the Database. If the Database goes offline, the App Server cannot verify any login — even though the App Server itself is perfectly healthy.
The highest-value internal target
The App Server holds all authentication and authorisation logic. If an attacker compromises it, they can access every customer account. It is protected behind the Firewall, Load Balancer, and Web Server — but remains the prime target.

Internal Secure Network
Customer Database
All account records · balances · transactions
The crown jewel. Every account record, every password hash, every transaction. A breach here exposes every NorthBridge customer. Protected behind the full 7-component chain.
Database Concepts
1 / 3
What the Database holds
Every account record. Every balance. Every transaction. This is the complete financial picture of every NorthBridge customer — all in one place.
The crown jewel — deepest in the chain
Six layers of components stand between the public Internet and the Database. An attacker must breach every one of them to reach it. That is by design.
Failure propagation — one cause, six steps
The Database going offline triggers a cascade through every dependent component. The error Rahul sees is at step 6. The cause is at step 1. Engineers must trace the chain.
